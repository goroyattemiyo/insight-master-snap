# 既存コード監査

## 対象
- `screen_dashboard.html`
- `screen_analytics.html`
- `docs/BACKLOG.md`

## 全体評価
既存コードは「流用素材」としては有用だが、Android向けMVPの土台としてはそのまま使わない方がよい。

### 理由
- 既存コードは多機能で、MVP範囲を大きく超えている
- 1ファイルがかなり肥大化している
- API / GAS / 自動更新 / AI機能など依存が強い
- 今回の `InsightMasterforAndroid` の主軸である「API不要・スクショ1枚・初心者向け」と思想がややズレる

## 推奨方針
**設計は新規、使えるロジックや観点だけ移植する。**

---

## A. 残す

### 1. 指標の見せ方
- ビュー
- フォロワー
- いいね
- 返信
- リポスト
- 引用
- リンククリック

主要指標の整理はAndroid版でも参考になる。

### 2. 分析の文章構造
数値をそのまま出すのではなく、意味を言語化する思想は新アプリでも活かせる。

### 3. 投稿分析の観点
- 投稿パフォーマンス一覧
- ER
- 時間帯
- バズパターン
- 上位投稿 / 下位投稿比較

分析軸そのものは有用。

### 4. 状態の持ち方の考え方
`App.state` ベースの状態管理は、新実装での状態分割の参考になる。

### 5. バックログの課題認識
- `screen_dashboard.html 分割`
- `screen_analytics.html 分割`
- `PWA対応`

既存コードの問題意識は妥当。

---

## B. 捨てる

### 1. 巨大な画面ファイル構造
`screen_dashboard.html` と `screen_analytics.html` は責務が集まりすぎている。
新アプリでは画面ごと・機能ごとに分割する。

### 2. API前提の導線
今回のMVPはスクショ解析型が主軸なので、そのまま持ち込まない。

### 3. 競合分析
Grounding検索、比較分析、競合記録などはMVP外なので初期版では捨ててよい。

### 4. 週次レポート、自動更新、自動記録
初期版では重すぎるため削る。

### 5. デモモードや高度な演出
初期版では不要。理解速度を優先する。

---

## C. 要確認

### 1. AI分析ロジック本体
画面側から呼ばれている生成ロジックが、どこまで再利用可能か確認が必要。

### 2. データ保存まわり
既存保存方式がGASや既存バックエンド依存なら、Android版では保存方式を再設計する必要がある。

### 3. ERやスコアの計算ロジック
計算式は再利用価値があるが、スクショ抽出データ前提で見直しが必要。

### 4. UIコンポーネントの再利用
カード構造などは参考にできるが、Android向けに簡略化が必要。

---

## 新アプリに移植してよいもの
- 主要指標の考え方
- ERなどの分析観点
- 良い点 / 弱い点 / 次の一手 の文章構造
- 投稿比較やバズ分析の発想
- 履歴画面の情報整理
- 状態管理の分け方の発想

## 新アプリで作り直すべきもの
- ホーム画面
- 結果画面
- 数値修正画面
- 履歴画面
- OCR → 抽出 → 修正 → 再解析 の流れ
- Android向けのファイル選択導線
- ブランド名と文言
- 画面構成の分割

## 初期実装の優先順位
1. スクショ選択
2. SNS選択
3. OCR / 数値抽出
4. 数値修正
5. 結果表示
6. 履歴保存

## 会議で決めるべきこと
- 既存コードから何を移植対象にするか
- AI分析ロジックを流用するか新規化するか
- 保存方式をどうするか
- UIはどこまで参考デザインを活かすか
- 競合分析などMVP外機能を完全に切るか
